# Socket Server 2.0

面向当前主流互联网用户设备，包括现代浏览器及移动装置，我们设计了此版本`Socket Server`。以期在可预见未来，为额度拍提供健壮的前端通讯服务。

## 设计要点

### 使用[WebSocket](https://tools.ietf.org/html/rfc6455)通讯协议

`WebSocket`是一种在服务器与客户端双向通讯的技术，随着HTML5的普及，现代浏览器基本上都已经原生支持`WebSocket`了，

基于`TLS`，`WebSocket`可以实现客户端到服务端双向安全传输。与访问一般的`HTTPS`页面相比，是相同的加密技术，消耗的计算资源，安全强度没有差别。

并且，该技术已在上一版本`Socket Server`实验性支持，基础设施方面没有障碍，因此技术细节不在此展开。

### 弃用Quotabid Socket协议

即用于上一版本`Socket Server`和`Flash`客户端交互，直接构建于TCP层上的消息/业务协议。

由于`Flash`技术即将走入历史，继续支持已无必要。

同时其也存在安全性缺陷，无法有效修复，也是重要因素。
- 通讯加密基于全局唯一对称密钥，由于密钥可以在客户端中取得，因此所有通讯流量可视作明文的。这是非常危险的，只是由于业务局限性，没有成为攻击点。

缺乏灵活的头部数据定义能力，对于功能扩展造成严重阻碍，影响项目质量。
- 需要添加业务无关数据时，只能添加到消息体中。对于JSON格式消息体来说，相对容易，增加一些多余字段，业务处理时需要特别排除。对于非JSON格式消息体则变成不可能。

- 或者在消息类型(如`1-1`)之后定义固定长度的字节位表示特定的字段值，若后续字段增加，则产生了一个新的消息协议版本，并且是业务强关联的，维护难度急剧增加。

### 使用[STOMP](http://stomp.github.io/stomp-specification-1.2.html)消息协议

具体到本项目，使用[STOMP Over WebSocket](http://jmesnil.net/stomp-websocket/doc/)。

以下内容摘录自[《STOMP 客户端 API 整理》](https://blog.csdn.net/jqsad/article/details/77745379)。

>## 介绍
>
>STOMP(Simple Text-Orientated Messaging Protocol) 面向消息的简单文本协议
>
>WebSocket是一个消息架构，不强制使用任何特定的消息协议，它依赖于应用层解释消息的含义；
>
>与处在应用层的HTTP不同，WebSocket处在TCP上非常薄的一层，会将字节流转换为文本/二进制消息，因此，对于实际应用来说，WebSocket的通信形式层级过低，因此，可以在 WebSocket 之上使用 STOMP协议，来为浏览器 和 server间的 通信增加适当的消息语义。
>
>如何理解 STOMP 与 WebSocket 的关系：
>1. HTTP协议解决了 web 浏览器发起请求以及 web 服务器响应请求的细节，假设 HTTP 协议 并不存在，只能使用 TCP 套接字来 编写 web 应用，你可能认为这是一件疯狂的事情；
>2. 直接使用 WebSocket（SockJS） 就很类似于 使用 TCP 套接字来编写 web 应用，因为没有高层协议，就需要我们定义应用间所发送消息的语义，还需要确保连接的两端都能遵循这些语义；
>3. 同 HTTP 在 TCP 套接字上添加请求-响应模型层一样，STOMP 在 WebSocket 之上提供了一个基于帧的线路格式层，用来定义消息语义；
>
>## STOMP帧
>
>STOMP帧由命令，一个或多个头信息、一个空行及负载（文本或字节）所组成；
>
>其中可用的COMMAND 包括：
>
>```
>CONNECT、SEND、SUBSCRIBE、UNSUBSCRIBE、BEGIN、COMMIT、ABORT、ACK、NACK、DISCONNECT；
>```
>
>例：  
>发送消息
>
>```
>SEND
>destination:/queue/trade
>content-type:application/json
>content-length:44
>
>{"action":"BUY","ticker":"MMM","shares",44}^@
>```
>
>订阅消息
>
>```
>SUBSCRIBE
>id:sub-1
>destination:/topic/price.stock.*
>
>^@
>```
>
>服务器进行广播消息
>
>```
>MESSAGE
>message-id:nxahklf6-1
>subscription:sub-1
>destination:/topic/price.stock.MMM
>
>{"ticker":"MMM","price":129.45}^@
>```

而上一版本`Socket Server`的`WebSocket`通道实现中，消息格式定义非常原始，不得不将几乎所有信息塞入一个`JSON`对象中，导致各处理环节无法有效拆分。

本着不重新发明轮子的思路，我们使用`STOMP`协议使交互变得简单、清晰，就像`HTTP`协议一样，而且还有双向通讯能力。

### 防抵赖

这里指的是防用户抵赖，即关键性指令必须能够证明是用户发出的，并且相关消息需要存档以备查询验证。用户登录验证，可以阻止未授权用户访问系统。但系统自身是否伪造了数据，如何自证清白？

可以抓取所有通讯流量，证明服务端程序没有作弊行为。这是我们已经在做的，但也存在一些弱点。
- 效率低

    通讯流量记录占用大量存储空间，验证过程需要扫描所有这些海量数据。
- 强度低

    用户可能不承认通讯流量记录，毕竟是由服务提供方提供的。类似于银行柜面每一笔交易，都需要用户签名，否则可以不承认。

考虑两种解决方案

- 消息签名

    用户首先在客户端生成密钥对，然后需要通过第三方权威机构，将公钥与真人绑定认证。

    后续用户发送的消息，某些关键操作，必须包含电子签名信息。

    第三方权威机构，可以使用活体认证。同时保留相关认证数据，以便证明绑定关系。

    这种方案适用于固定、自有设备的用户，特别是家庭电脑或者个人手机。

    对于使用公共设备的用户，特别是网吧等场所。存在两个主要问题：
    1. 认证麻烦，每次拍卖都要进行一次认证。
    2. 风险高，用户的认证信息留在设备上，可能会被盗用。

- 一次性令牌

    用户进行某些关键操作时，需要提交额外的认证信息。

    有点类似于，向注册的用户手机号发送验证码，用户提交请求时同时包含收到的验证码。

    但是如果短信内容是服务端生成的，那么该验证码服务端也是知道的，对于留存的用户请求消息，并不能说明真实来源。

    因此需要有一个第三方一次性令牌服务，即服务端调用该一次性令牌服务，指定发送手机号，以及附加信息，但不包含验证码。一次性令牌服务生成验证码，向指定手机号发送短信。服务端收到用户请求时，利用其提交的验证码，调用一次性令牌服务验证有效性。
	
	存在主要问题是，对于出价操作，时间不允许；用在登录阶段，这样证据强度就变低了。
	
	可以与消息签名结合，即通过将公钥与手机号绑定认证，间接实现公钥与真人绑定认证。
